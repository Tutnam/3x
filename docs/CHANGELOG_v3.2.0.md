# Журнал доработок VPN-бота (миграция на 3x-ui v3.2.0)

Документ описывает все изменения, внесённые в проект в ходе перевода бота со
старой панели **3x-ui 2.9.4** на новую **3.2.0**, а также сопутствующие
улучшения деплоя, управления подписками и обслуживания БД.

---

## 1. Миграция логики работы с панелью на v3.2.0

### Проблема
В версии 2.9.4 клиенты хранились **внутри инбаунда**
(`inbound.settings.clients[]`): чтобы добавить/удалить/изменить клиента, нужно
было прочитать весь инбаунд, поправить JSON-массив и записать инбаунд целиком.

В версии 3.2.0 модель данных нормализована:
- **клиент — отдельная сущность** (таблица `clients`: `email`, `uuid`, `subId`,
  `flow`, `totalGB`, `expiryTime`, `enable`, `tgId` …);
- к клиенту **привязываются инбаунды** через таблицу `client_inbounds`
  (many-to-many) — это и есть «подключения в профиле клиента»;
- появились прямые эндпоинты управления клиентами.

### Что сделано (`src/functions.py`)
Полностью переписан класс `XUIAPI` под новые эндпоинты v3.2.0, при этом
**сохранён публичный контракт** функций (чтобы `handlers.py` / `app.py` /
`subscription_server.py` почти не менялись).

| Операция | Эндпоинт v3.2.0 |
|---|---|
| Создать клиента | `POST /clients/add` `{client, inboundIds}` |
| Получить клиента | `GET /clients/get/{email}` |
| Обновить клиента | `POST /clients/update/{email}` (плоское тело, замена целиком) |
| Удалить клиента | `POST /clients/del/{email}?keepTraffic=0` |
| Ссылки клиента | `GET /clients/links/{email}` |
| Трафик клиента | `GET /clients/traffic/{email}` |
| Онлайн | `POST /clients/onlines` |
| Список клиентов | `GET /clients/list` |

Ключевое улучшение: **конфиг-ссылки теперь генерирует сама панель**
(`/clients/links/{email}`) — для всех привязанных инбаундов и протоколов
(reality/ws/xhttp/trojan, RU+LV). Ручная сборка `generate_vless_url` оставлена
только как аварийный fallback.

### Сервер подписок (`src/subscription_server.py`)
Переписан: вместо ручной сборки одного URL отдаёт **base64 готовых ссылок
панели** по email пользователя (стандартный формат подписки — несколько строк).

### Хендлеры (`src/handlers.py`)
Статические профили теперь показывают/хранят ссылки от панели (несколько строк).

### Проверка
Сквозной smoke-тест на живой панели: создание → привязка ко всем инбаундам →
получение ссылок → enable/disable → трафик → удаление. Существующие пользователи
(email вида `user_<tgid>_<rand>`) уже были мигрированы панелью и продолжают
работать.

## 2. Динамический список инбаундов

### Идея
Чтобы при добавлении нового сервера/инбаунда не править конфиг, бот при создании
клиента **сам запрашивает актуальный список инбаундов** у панели и привязывает
клиента ко всем клиентским инбаундам.

### Что сделано
- `src/functions.py`: метод `get_all_inbound_ids()` — `GET /inbounds/list`,
  фильтрует по клиентским протоколам (vless/vmess/trojan/shadowsocks), отсекая
  служебные (dokodemo/wireguard/http/socks).
- Логика выбора инбаундов при создании клиента:
  1. если `INBOUND_IDS` задан в `.env` — используется он (явное закрепление);
  2. иначе — **авто-режим**: актуальный список с панели;
  3. fallback — единственный `INBOUND_ID`.
- `src/config.py` / `.env`: добавлена переменная `INBOUND_IDS` (по умолчанию
  пуста = авто-режим).

Проверено на живой панели: после переконфигурации сервера список инбаундов
подхватился автоматически.

## 3. Скрипт установки и запуска (`start-bot.sh`)

Файл был затёрт; переписан в полноценный установщик для Ubuntu-сервера.

Шаги скрипта:
1. `apt update && apt upgrade` — обновление пакетов системы;
2. установка system-зависимостей (`python3`, `venv`, `pip`) и `screen`
   (с проверкой наличия);
3. создание виртуального окружения `.venv` (если ещё нет);
4. обновление `pip`;
5. установка зависимостей из `requirements.txt`;
6. **интерактивный ввод `BOT_TOKEN` (токен бота) и `ADMINS` (id админа)** с
   записью в `src/.env`;
7. **настройка автозапуска после перезагрузки** через `cron @reboot`;
8. запуск бота в screen-сессии `mybot`.

Особенности:
- путь берётся из расположения самого скрипта (без хардкода `/root/3x`);
- идемпотентность (повторный запуск безопасен; screen-сессия перезапускается);
- автоматическое использование `sudo`, если запущен не от root;
- ввод `BOT_TOKEN`/`ADMINS`: показ текущего значения с возможностью оставить,
  защита от пустого ввода, пропуск в неинтерактивном режиме;
- `@reboot`-запись в crontab идемпотентна (по тегу `# vpn-bot-autostart`),
  не плодит дубли и не трогает чужие задачи.

> ⚠️ Автозапуск через `cron @reboot` поднимает бота напрямую (venv + `app.py`),
> **без** повторного `apt upgrade`, чтобы загрузка сервера не зависала на
> apt-локах.

## 4. Управление сроком подписки (expiryTime)

### Проблема
Клиент в панели создавался с `expiryTime: 0` (безлимит) — в панели время
выглядело «без ограничения», срок контролировал только сам бот.

### Что сделано
В v3.2.0 у клиента есть собственное поле `expiryTime` (Unix-время в
**миллисекундах**) — бот стал его проставлять:
- `src/functions.py`: `_create_client(..., expiry_ms)` пишет `expiryTime` при
  создании; `set_client_subscription(email, expiry_ms, enable)` синхронизирует
  срок и состояние одним запросом;
- `src/handlers.py`: при создании профиля передаётся
  `expiry_ms = subscription_end` (триал/оплата сразу отражаются в панели).

## 5. Панель как источник правды по времени

### Идея
Не дублировать учёт времени: панель сама следит за `expiryTime` и отключает
клиента по истечении. Бот **читает** срок из панели и **добавляет** время
запросом в панель при оплате/изменении админом.

### Что сделано
- `src/functions.py`:
  - `get_clients_list()` — `GET /clients/list` (только чтение; ничего не пишет);
  - `add_time_by_email(email, seconds)` — read-modify-write `expiryTime`
    (продление истёкшего считается от «сейчас», отрицательное = списание).
- `src/app.py`: фоновая задача `check_subscriptions` заменена на
  **`sync_subscriptions`**:
  - раз в час читает `/clients/list` и **зеркалит** `expiryTime` →
    `subscription_end` в `users.db` (ручные правки в панели тоже
    подхватываются);
  - шлёт **предупреждение за 24 часа** до истечения;
  - **бот больше не отключает клиентов** — это делает панель.
- `src/handlers.py`: единый хелпер **`grant_time(telegram_id, seconds)`**:
  - есть профиль → меняет срок **в панели** + зеркалит в БД;
  - нет профиля → правит только БД (применится при «Подключить»);
  - используется при **оплате** и **админ ±время**;
  - убраны push-синхронизация и мусорный флаг `disabled`.

Проверено живым тестом: триал ровно 3 дня → активная подписка → истечение
(панель отключает) → продление → 24ч-уведомление шлётся один раз.

## 6. Кнопка «➕ время всем активным»

В админ-меню добавлена возможность добавить время **сразу всем пользователям с
активной подпиской**.

Поток: админ-меню → «➕ время всем активным» → ввод `Месяцы Дни Часы Минуты` →
подтверждение с количеством → применение через `grant_time` каждому активному →
отчёт `Добавлено / Ошибок / Всего`.

«Активные» = `get_all_users(with_subscription=True)` (`subscription_end > now`).
Истёкшие и без подписки не затрагиваются. Проверено на временной БД.

## 7. Диагностика «воскрешающихся» клиентов

### Симптом
Удалённый в панели клиент (`user_<id>_<old>`) появлялся снова.

### Разбор
- В `users.db` бот хранит **один** профиль на пользователя (email в
  `vless_profile_data`). Бот знает только актуального клиента и **не воскрешает**
  старых.
- Причина оказалась на стороне инфраструктуры панели: клиент присутствовал на
  **нескольких серверах/нодах** (mind RU + FI + др.), был зашит в
  `settings`-JSON инбаундов и привязан к нодам. Удаление на одном сервере
  оставляло копии на других, и они синхронизировались обратно.

### Решение
Удалять клиента **на всех серверах/нодах**. После этого клиент перестал
появляться. Бота для этого останавливать не требуется.

> Рекомендация на будущее: добавить в админку кнопку «🗑 Удалить профиль
> пользователя», которая делает штатный API-delete панели и чистит запись в
> `users.db` — чтобы удаление шло в один шаг и без рассинхрона. (Пока не
> реализовано.)

## 8. Чистка БД `users.db`

После обновления логики проверены столбцы таблицы `users` на актуальность.

| Столбец | Статус |
|---|---|
| `id`, `telegram_id`, `full_name`, `username`, `subscription_end`, `vless_profile_data`, `subscription_id`, `is_admin`, `notified` | ✅ используются |
| `registration_date` | 🟡 метаданные, оставлен |
| **`vless_profile_id`** | 🔴 рудимент старой логики — **удалён** |
| `static_profiles.created_at` | 🟡 метаданные, оставлен |

### Что сделано
- `src/database.py`: убран столбец `vless_profile_id` из модели `User`.
- `src/cleanup_db.py`: разовый идемпотентный скрипт чистки —
  1. бэкап `users.db` → `users.db.bak_cleanup`;
  2. `ALTER TABLE users DROP COLUMN vless_profile_id` (SQLite ≥ 3.35; иначе
     безопасный пропуск);
  3. удаление мусорного флага `disabled` из JSON `vless_profile_data` у всех
     пользователей (легаси от до-panel-authoritative логики).

Запуск (с остановленным ботом):
```bash
cd /home/boss/cursor_project/3x
.venv/bin/python src/cleanup_db.py
```

## Модель работы после миграции (кратко)

- **Панель** — единственный enforcer срока (отключает клиента в момент
  `expiryTime`) и генератор конфиг-ссылок.
- **Бот** — читает срок из панели и зеркалит в `users.db` (для меню/списков),
  добавляет время запросом в панель при оплате/админке, создаёт клиента при
  «Подключить» (привязывая ко всем актуальным инбаундам).
- **Триал 3 дня** и предоплата без профиля живут в `users.db` и переносятся в
  панель при создании клиента.
