# Запуск бота в Docker

Контейнеризуется **только бот**. Панель 3x-UI остаётся как есть — бот ходит в неё
по `XUI_API_URL` по сети.

## Предпосылки на хосте
- Установлены Docker и плагин `docker compose`.
- Рядом с `docker-compose.yml` (в корне проекта) лежат:
  - `src/.env` — конфиг (секреты);
  - `users.db` — живая база (монтируется в контейнер);
  - папка `backups/` (создастся сама, если её нет).

## Перенос на новый сервер
Порядок важен: `src/.env` и `users.db` нужно положить **до** `deploy.sh`.

```bash
# 1. Принести код
git clone <repo> && cd 3x        # или scp/rsync всю папку проекта

# 2. Принести конфиг (его НЕТ в git — .gitignore!)
scp old-server:/path/3x/src/.env  src/.env

# 3. Принести базу с юзерами (если переносишь, а не стартуешь с нуля)
scp old-server:/path/3x/users.db  users.db

# 4. Одна команда
bash deploy.sh
```

Почему именно так:
- **`src/.env` обязателен.** Он в `.gitignore`, `git clone` его не приносит, а без
  него `deploy.sh` сразу упадёт (там BOT_TOKEN, API-токен панели и т.д.).
- **`users.db` копировать ДО запуска.** `deploy.sh` создаёт пустой `users.db` и
  поднимает контейнер, который держит файл открытым. Если скопировать базу
  поверх уже запущенного контейнера — это запись в открытый файл → порча SQLite
  (или контейнер продолжит работать со старой пустой базой). Если уже запустил —
  сначала `docker compose down`, потом копируй, потом `up -d`.
- Шаг 3 можно пропустить — тогда стартуешь с **чистой** базой (бот создаст её сам,
  юзеры регистрируются заново). Перенесённая `users.db` уже мигрирована —
  `create_all` её не трогает, миграция не нужна.

Если это **полный переезд** (не только бот):
- Панель по `XUI_API_URL` — публичный домен, бот достучится без настройки.
- **Подписочный reverse-proxy/домен** (`https://.../FPtzhP2oo1` → `127.0.0.1:35635`)
  нужно настроить на новом сервере заново, иначе у юзеров не обновятся подписки.

## Запуск одной командой (рекомендуется)
```bash
bash deploy.sh
```
Скрипт сам: проверит/установит Docker и docker compose, запустит демон, создаст
пустой `users.db` при чистом старте (чтобы том смонтировался файлом, а не папкой),
поднимет контейнер и дождётся `/health`. Идемпотентный — повторный запуск
просто пересоберёт и обновит. Запускать от root или через sudo.

## Первый запуск вручную
```bash
# из корня проекта
docker compose up -d --build
docker compose logs -f bot      # смотреть логи
```

⚠️ **Сначала останови старый запуск** (screen/systemd/cron), чтобы две копии бота
не работали с одной `users.db` одновременно — иначе блокировки SQLite и гонки.
```bash
screen -S <session> -X quit     # если бот крутится в screen
```

## Миграция БД
Текущая `users.db` уже мигрирована — для неё ничего делать не нужно.
Для **новой/чистой** базы выполнить один раз (бот создаст таблицы сам в init_db,
а доп. колонки добавит миграция):
```bash
docker compose run --rm bot python migrate_features.py
```

## Обновление кода
```bash
git pull            # или внеси правки
docker compose up -d --build
```
Контейнер пересоздаётся, но `users.db` и `backups/` лежат на хосте (тома) и
сохраняются.

## Полезные команды
```bash
docker compose ps               # статус + health (healthy/unhealthy)
docker compose restart bot
docker compose down             # остановить и удалить контейнер (данные на хосте целы)
docker compose logs --tail=100 bot
```

## Персистентность (важно)
Файловая система контейнера эфемерна. На хост вынесены **тома**:
- `./users.db:/app/users.db` — без этого обновление бота стёрло бы всех юзеров;
- `./backups:/app/backups` — без этого суточные авто-бэкапы пропадали бы при
  каждом пересоздании контейнера.

Нюансы:
- `users.db` монтируется как **файл** — он должен существовать на хосте до первого
  `up` (иначе Docker создаст вместо него папку). У нас он есть.
- SQLite создаёт временный `users.db-journal` рядом с базой внутри контейнера —
  это нормально (живёт только в рамках транзакции).
- Права: контейнер пишет под своим UID; папка `backups/` и файл `users.db` должны
  быть доступны ему на запись. Если авто-бэкап молча не создаётся — смотри права.

## Healthcheck
Бот поднимает `GET /health` на подписочном сервере (порт `SUBSCRIPTION_PORT`,
дефолт 35635) — отдаёт `{"status":"ok"}`. Docker дёргает его каждые 30с; при
3 неудачах подряд контейнер помечается `unhealthy` (видно в `docker compose ps`).
Внешний мониторинг тоже может опрашивать этот эндпоинт.

## Сеть до панели
`XUI_API_URL=https://rus.vlip.ru.net` — публичный домен, контейнер достучится до
него без доп. настройки. Если панель переедет на localhost того же хоста — см.
комментарий в конце `docker-compose.yml` (host.docker.internal или network_mode: host).

## Подписочный сервер и reverse-proxy
Сервер слушает 35635 внутри контейнера и проброшен на `127.0.0.1:35635` хоста —
внешний reverse-proxy продолжает отдавать `https://rus.vlip.ru.net/FPtzhP2oo1`
как раньше. Если меняешь `SUBSCRIPTION_PORT` в `.env` — поправь маппинг портов в
`docker-compose.yml`.
